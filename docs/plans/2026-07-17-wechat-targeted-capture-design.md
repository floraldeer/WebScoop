# 微信视频号指定 URL 捕获设计

## 问题

现有注入脚本把 `encfilekey -> media/decode_key` 候选表保存在页面内。桌面微信会同时使用主页、详情页和额外 WebView，候选响应与实际视频请求可能出现在不同 JavaScript 上下文。日志中的连续 `flush-miss` 证明这种状态隔离会使后续视频无法配对；单页 `activeKeys` 只能修复事件先后顺序，不能修复跨页面丢配。

状态机还有一个独立问题：真实媒体合并到 `infoOnly` 占位项时没有回填 `decodeKey`，导致已捕获卡片仍可能下载加密字节。

## 方案

新增主进程级微信捕获协调器。注入脚本只负责从 `finderH5ExtTransfer` 提取候选媒体，并上报所有 `encfilekey`、`decode_key`、对象 ID、作者、描述及“是否为当前详情对象”。hoxy 在 `finder.video.qq.com` 请求阶段上报实际活动的 `encfilekey`。协调器使用全局 Map 双向配对，因此候选先到、请求先到、跨 WebView 到达都能得到相同结果，并通过 TTL 清理长期状态。

粘贴视频号分享 URL 时，主进程注册一个目标任务，保存原始 `shareUrl`、描述和作者。目标存在期间，只接受与目标元信息一致的当前详情对象；匹配成功后，捕获结果携带原 `shareUrl` 返回前端。前端优先按 `shareUrl` 合并占位项，并完整回填 URL、高清 URL、大小、`decodeKey` 和解密标志。目标匹配失败时不把相邻预加载视频写入该任务。

## 边界与验证

改动仅覆盖视频号注入回传、视频号 CDN 特判和视频号 IPC。其他平台解析器、通用 CDN 捕获、下载队列不改。单元测试覆盖候选先到、请求先到、目标过滤、重复请求和状态过期；最后运行 Electron 与 Web 构建。
